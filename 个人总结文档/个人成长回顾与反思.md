# DreamAIChat · 个人成长回顾与反思

> 作者：邢浩哲  
> 视角：个人能力成长 / 职业素养提升

---

## 1. 从“写代码”到“做产品”的角色转变

在这个项目之前，我对自己的定位更多是“写 Android 代码的人”：  
- 关注点主要在 Activity、Fragment、Adapter 的实现；  
- 对后端、架构、部署等话题只停留在“能用就行”的层面。

通过 DreamAIChat，我第一次完整地扮演了每一个参与项目实现的角色：
- 产品经理：自己定义功能优先级、体验细节和迭代路线图；  
- 技术人员：同时关心前端架构、后端接口设计和部署方案；  
- Prompt Engineer：通过高质量指令驱动 AI 参与编码与文档编写。

这一角色转变，让我对“工程师”的理解从“写多少行代码”转向“对产品全链路负责”。

---



## 2. 技术维度的成长

### 2.1 Android 端：更系统地理解 MVVM / Room / Retrofit

在实践过程中，我不再只是机械地照抄教程，而是刻意思考：
- 为什么登录模块用 MVI 更合适？  
- 聊天与历史为什么改为 MVVM + Repository 更好维护？  
- Room 的外键、级联删除在实际业务中会带来什么副作用？

几个关键收获：
- 学会用 ViewModel + LiveData 管理状态，而不是在 Fragment 里堆逻辑；  
- 通过一次真实的“级联删除事故”，对 Room 的约束和迁移策略有了深刻教训；  
- 对 Retrofit + OkHttp 的超时时间、日志拦截器等配置理解更具“工程思维”。

### 2.2 后端与 DevOps：从“黑盒 API”到“自己搭服务”

过去我习惯把后端看成一个固定地址的黑盒。  
在本项目中，我亲手参与/理解了：
- Spring Boot 工程的目录结构、Controller/Service/Repository 的分工；  
- JWT 登录认证的基本流程；  
- 如何通过配置文件与环境变量切换不同模型提供方；  
- 如何把后端从本地移植到云服务器，并通过公网 IP 服务移动端。

这些经历让我明白：  
> 当你能自己搭起一个后端服务时，前端开发会多一种“可以自己解决”的底气。

## 3. 与 AI 协作能力的提升

### 3.1 Prompt 写作从“自然语言”到“需求规格书”

一开始，我只是用自然语言描述需求，效果参差不齐。  
逐步实践后，我建立了一套适合自己的 Prompt 模板：
- 明确角色（你是 Android 工程师 / 后端工程师 / 技术文档作者）；  
- 给出上下文（当前文件路径、已有实现、文档约束）；  
- 指明输出格式（代码片段 / Markdown 文档 / 表格）；  
- 强调约束条件（不能改哪些、必须更新哪些文档）。

结果是：
- AI 生成的代码/文档越来越贴近我想要的风格；  
- 迭代次数明显减少，我的精力更多用在审核与微调，而不是反复重写。

### 3.2 保持主导权：不做 AI 的“复制粘贴员”

我在项目中刻意坚持几个原则：
- 每一段 AI 提供的代码，至少要知道**为什么这么写**；  
- 关键逻辑（如 Session 管理、数据表设计）不完全交给 AI 决策，而是由我拍板；  
- 出现 bug 时，先自己推理一遍，再借助 AI 工具验证和补充。

这让我在使用 AI 时既享受效率，又不会失去对代码的掌控感。

## 4. 工程素养与合作意识

### 4.1 文档意识：为“未来的我”和“潜在的协作者”写

项目中，我投入了相当多时间在写：
- 架构设计白皮书；  
- 部署指南；  
- 故障排查文档；  
- Prompt 实践报告。

一开始只是为了满足训练营要求，但写着写着，我发现这些文档真正的价值在于：
- 让未来的我可以“快速回忆起这段代码在想什么”；  
- 任何接手这个项目的人可以在 30 分钟内跑起来，而不用对着迷宫般的目录发呆；  
- Prompt 工程的经验可以迁移到后续项目中复用，而不是每次从零开始摸索。

### 4.2 质量意识：每一个 bug 都是可复用的教训

项目里出现的典型问题（如历史丢失、iconGravity 错误、资源路径不一致），我都：
- 找到根因；  
- 修复代码；  
- 在《故障排查及解决文档》中记录“错误表现→原因→解决→预防”；  
- 在后续开发中有意识地避免类似设计。

这让我从“修完就忘”的状态，逐步进入一种**积累经验库**的节奏。

---



## 5. 不足与改进方向

1. **自动化测试不足**：  
   - 目前项目仍主要依赖手测和日志分析；  
   - 未来有必要为关键用例（登录、发消息、历史加载）补充自动化测试。

2. **性能压测不系统**：  
   - 虽然增加了超时时间，但还没做过压力测试和长时间使用测试；  
   - 后续可以引入简单的压测工具或日志分析。

3. **团队协作场景还不够**：  
   - 本项目更多是“我 + AI”的协作；  
   - 未来希望能在多人协作团队中验证这些 Prompt 和工程化实践的可推广性。

---



## 6. 总结

在 DreamAIChat 项目中，我不仅完成了一个可以演示的 AI 对话 APP，更重要的是：
- 学会了如何从产品、架构、实现、文档多个维度统筹一件事；  
- 体会到与 AI 协作的真正价值：不是替我写代码，而是帮助我更快、更系统地把想法变成工程；  
- 建立了一套适合自己的工作方法：**明确目标 → 精准 Prompt → 审核与迭代 → 文档沉淀**。



**我想说:** 

**这次的字节训练营经历对我来说不仅仅是为了向优秀营员乃至直通实习终面而奋斗, 更在于这过程中挑战新的技术栈 , 学习新的开发知识 , 感受大厂的工程化项目开发全流程 , 更是认识了一众优秀的技术老师和小伙伴们，也让我更有信心面对未来在企业工作中更大规模、更加复杂的困难与挑战 !**
